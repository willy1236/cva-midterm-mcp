# 代理式 AI 固定限制機制研究計畫（修正版）

## 一、研究目的
本研究希望建立一套可重複套用於代理式 AI 的固定限制機制，讓系統在不同情境、不同任務與多代理協作下，仍能維持一致的行為邊界。研究重點不是追求模型自由度，而是提升可控性、可驗證性與可治理性，讓使用者可以明確知道 AI 只會在授權範圍內行動。

具體目標如下：
1. 建立一套以 Prompt 與設定檔為核心的限制框架，降低過度分析、跨界推論與不必要行為。
2. 驗證同一組限制規則能否跨情境穩定生效，而非只對單一任務有效。
3. 將信任機制落地為可觀察、可測試、可稽核的流程。

## 二、研究問題與假設
### 2.1 研究問題
1. 同一組固定限制 Prompt 與治理規則，是否能在不同情境維持一致邊界？
2. 固定限制框架是否能顯著降低越界行為、幻覺與工具濫用？
3. 多代理審查機制是否能提升輸出可靠性，而不造成不可接受的延遲與成本？

### 2.2 研究假設
1. 若限制規則結構化且可機器驗證，代理行為一致性會優於僅靠自由指令的做法。
2. 若加入引用查核、工具閘道與輸出驗證，錯誤資訊與格式失敗率會下降。
3. 若加入多代理審查，整體可信度會提升，但推理時間與成本會上升，需透過熔斷機制平衡。

## 三、研究範圍與非範圍
### 3.1 本研究涵蓋
1. 固定限制 Prompt 設計與配置化管理。
2. 多模組治理流程：身份邊界、輸出驗證、引用查核、工具權限、多代理審查、安全分類、成本熔斷。
3. 在多情境下的對照實驗與量化評估。

### 3.2 本研究不涵蓋
1. 基礎模型重訓或微調。
2. 完整企業資安稽核與法遵認證流程。
3. 所有 AI 安全問題的一次性解決方案。

## 四、系統化限制框架（七模組）
### 模組 1：行為指令與角色邊界
目的：定義代理身份、任務邊界與不可跨越底線。

核心機制：
1. 以 get_agent_profile(context_id) 注入情境化身份規則。
2. 以 fetch_constraint_config() 載入全域限制設定。

拒絕條件：當請求超出身份或任務邊界時，必須拒答或降級回覆。

### 模組 2：結構化輸出與驗證
目的：確保所有輸出可被下游系統穩定解析。

核心機制：
1. 以 validate_output_structure(data, schema_type) 驗證格式。
2. 使用 schema（如 Pydantic）檢查欄位完整性與型別。

拒絕條件：不符合 schema 的輸出不得進入下游流程。

### 模組 3：知識接地與引用查核
目的：限制回答必須有可追溯依據，降低幻覺。

核心機制：
1. 以 verify_citations(claims, sources) 比對主張與引用。
2. 強制引用可定位來源（頁碼、段落、資料庫 ID）。

拒絕條件：引用無法對應時，標記為「未證實」或拒絕輸出。

### 模組 4：外部工具與存取管控
目的：限制工具使用範圍，避免 API 濫用與敏感資料外洩。

核心機制：
1. 以 secure_tool_call(tool_name, arguments) 作為統一閘道。
2. 執行 Tool_Scope、存取層級與操作型別檢查。

拒絕條件：若唯讀工具被用於寫入行為，應即時阻斷並記錄。

### 模組 5：多代理審查機制
目的：用代理互審降低單點錯誤，建立交叉驗證。

核心機制：
1. 以 submit_for_peer_review(content, criteria) 送審。
2. 未通過審查之內容不得標記為授權輸出。

拒絕條件：審查不通過或審查結果不一致時，回到修正流程。

### 模組 6：內容安全分類與動態政策
目的：過濾違反倫理、法律或高風險內容。

核心機制：
1. 對輸出執行安全分類。
2. 依情境切換政策等級（一般、敏感、高風險）。

拒絕條件：命中高風險政策時直接阻擋，不進入回覆階段。

### 模組 7：資源與成本熔斷
目的：避免無限推理循環與失控的 API 成本。

核心機制：
1. 監控 token、呼叫次數、工具頻率與總延遲。
2. 超過門檻時啟動熔斷與降級策略。

拒絕條件：超出預算或延遲閾值時終止任務並回報原因。

## 五、實驗設計
### 5.1 情境設計
1. 情境 A：ESG 分析代理。
2. 情境 B：自動化程式開發代理。
3. 情境 C：一般知識問答代理（作為中性對照）。

### 5.2 實驗組別
1. Baseline：無固定限制，僅一般系統提示。
2. Prompt-only：僅套用固定限制 Prompt。
3. Full Framework：套用七模組治理流程。

### 5.3 測試樣本
1. 正常任務樣本：標準授權請求。
2. 邊界挑戰樣本：模糊授權、跨域要求。
3. 攻擊樣本：Prompt Injection、錯誤引用、工具濫用。

### 5.4 實驗流程
1. 初始化：載入情境設定與限制規則。
2. 執行：代理完成任務並觸發工具調用。
3. 驗證：格式、引用、安全與審查流程逐層檢查。
4. 結案：記錄治理稽核日誌與量化指標。

## 六、評估指標與成功標準
### 6.1 主要指標
1. 邊界一致性（同規則跨情境是否維持相同行為）。
2. 越界率（未授權操作比例）。
3. 幻覺率（無可驗證依據的主張比例）。
4. 引用可追溯率（可定位來源比例）。
5. 格式合格率（符合 schema 輸出比例）。
6. 工具誤用率（不符合 Tool_Scope 的呼叫比例）。
7. 審查通過率（一次通過比例與重審次數）。
8. 成本與延遲（平均 token、平均回應時間、峰值資源）。

### 6.2 成功門檻（可依實驗調整）
1. Full Framework 相較 Baseline：越界率與幻覺率需明顯下降。
2. 格式合格率與引用可追溯率需維持高水位。
3. 成本與延遲增加需在可接受範圍內。

## 七、威脅模型與失效模式
主要威脅如下：
1. Prompt Injection：誘導代理忽略限制規則。
2. Citation Fabrication：引用看似完整但不可追溯。
3. Tool Abuse：以合法工具執行未授權行為。
4. Review Collusion：多代理審查出現同質偏誤。
5. Infinite Reasoning：陷入長鏈推理造成資源耗盡。
6. Policy Conflict：多政策同時命中導致決策不一致。

對策原則：所有拒絕、降級與放行決策都需可記錄、可追查、可重現。

## 九、預期貢獻
1. 提供一套可移植、可重複、可量化驗證的代理式 AI 限制框架。
2. 建立從 Prompt 到治理流程的實作路徑，縮短研究到落地的距離。
3. 讓企業在導入多代理系統時，有一套可解釋、可稽核、可持續調整的信任機制。
